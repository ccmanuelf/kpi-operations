# KPI-Operations — PR-2: C5 Collapse Schema Evolution to Alembic — Design

**Date:** 2026-07-09
**Status:** Design approved (user, 2026-07-09) — pending implementation plan
**Parent:** `2026-06-26-kpi-operations-production-deployment-design.md` (PR-2 section; all its locked decisions carry over). PR-1 (MariaDB portability, #128 `90da6cb`) is merged — `Base.metadata` already carries the corrected column types.
**Scope:** Make Alembic the ONLY schema-evolution mechanism — app startup, tests, demo seeding, and CI all build schema exclusively through migrations. Remove every other schema path.

---

## Context (verified against the codebase 2026-07-09)

- `backend/alembic/` is scaffolded and clean: `env.py` sources `settings.DATABASE_URL` (single URL source), imports all models, sets `compare_type=True` and `render_as_batch` only on SQLite. But the sole revision, `versions/001_baseline.py`, is a **no-op** (`upgrade()`/`downgrade()` are `pass`).
- Schema really comes from `create_all`, in **five places**: lifespan `init_schema()` (`bootstrap/lifecycle.py:122`), lifespan `create_capacity_tables()` (`lifecycle.py:181`, duplicating tables already in `Base.metadata`), the demo reseed drop/recreate path (`lifecycle.py:93-95`), the demo seeder's own-session path (`scripts/init_demo_database.py:335-341`), and three test fixtures in `tests/conftest.py` (module-global `get_test_engine`, function-scoped `db_engine`, function-scoped `transactional_db` — which drops/creates per test despite a docstring claiming rollback semantics).
- `docker-entrypoint.sh` runs `alembic upgrade head` (currently a no-op) with failures swallowed (`|| WARN`), plus an inline capacity-tables python block. Local dev (`uvicorn --reload` per README) bypasses the entrypoint entirely.
- A runtime migration API (`routes/database_config.py`, `/api/admin/database/*`) can create schema and repoint the live DB; the frontend has a live Admin → Database page wired to all six endpoints (`stores/databaseConfigStore.ts`, route `/admin/database`).
- Offline tooling: `scripts/migrate_database.py` CLI + `db/migrations/` (`SchemaInitializer`, `DataMigrator`, `capacity_planning_tables.py`, orphaned `phase2_junction_tables.sql`) + ~1,280 lines of tests.

## Design

### 1. Real baseline migration
- Replace `versions/001_baseline.py` with a genuine autogenerated baseline from `Base.metadata` (`alembic revision --autogenerate` against a fresh DB, then reviewed/committed). `down_revision = None`. Must build identically on SQLite (batch mode) and MariaDB.
- **Empty-autogen-diff guard (deterministic):** a test builds a fresh DB via `command.upgrade(cfg, "head")` and asserts Alembic's metadata comparison against `Base.metadata` yields an **empty diff**. Runs always-on in the SQLite suite AND against live MariaDB in the `mariadb-portability` CI job. Any model/migration drift fails CI on both dialects.
- The `alembic_version` table becomes meaningful; future schema changes are new revisions (never edits to shipped ones).

### 2. Startup rewiring
- `bootstrap/lifecycle.py`: delete the `init_schema()` and `create_capacity_tables()` units. New **first** lifespan unit `run_startup_migrations()`: programmatic `command.upgrade(cfg, "head")`, **fatal on failure**, gated by `RUN_MIGRATIONS_ON_STARTUP` (new setting, default `true`).
- Demo reseed path (`_auto_seed_demo_data`): `drop_all` → **`command.upgrade(cfg, "head")`** (was `create_all`). The DEMO_MODE-first guard stays exactly as is.
- `scripts/init_demo_database.py`: remove its `create_all` + `create_capacity_tables()` calls. When called with an existing session (in-app path), schema presence is the caller's responsibility (lifespan/entrypoint migrated already). The standalone `__main__`/owns-session path stays usable and runs `command.upgrade(cfg, "head")` before seeding.
- `backend/scripts/docker-entrypoint.sh`: remove the inline capacity-tables python block; `alembic upgrade head` becomes **fatal** (no `|| WARN`) when `RUN_MIGRATIONS=true` (existing env var, unchanged semantics).
- Render (unchanged config, verified safe): `RUN_MIGRATIONS=true` entrypoint migrate + in-app flag default true → idempotent no-op double-run under single-worker uvicorn.

### 3. Deletions (single source of truth)
- Delete: `backend/db/migrations/` (entire package), `backend/scripts/migrate_database.py`, `backend/tests/test_db/test_migration_integration.py`, `backend/tests/test_scripts/test_migrate_database.py`.
- **No-`create_all` guard (deterministic):** a test scans `backend/` sources (excluding `alembic/` and this guard's allowlist, initially empty) and fails on any `create_all` occurrence — the old mechanism cannot creep back.

### 4. Runtime API + frontend (D1)
- `routes/database_config.py`: keep GET `/status`, GET `/providers`; delete POST `/migrate`, POST `/test-connection`, GET `/migration/status`, GET `/full-status`, the `run_migration` background task, and provider-repointing state writes.
- Frontend: `databaseConfigStore.ts` pruned to the two surviving calls; `/admin/database` page reduced to a read-only status panel; unused components removed; **i18n keys pruned in both locales** (en+es) so the `referenced-keys` and `no-raw-text` CI gates stay green.
- `tests/test_db_routes/test_database_config.py` pruned to the surviving endpoints; frontend store/page tests updated.

### 5. Test schema — template DB (D4)
- Session-scoped fixture: `alembic upgrade head` ONCE into a file-based template SQLite DB (tmp path).
- All three existing mechanisms become thin **clones** of the template via SQLite's backup API, preserving their scopes and isolation semantics: module-global engine (used by `test_client` + token-blacklist autouse), function-scoped `db_engine` (fresh clone per test), `transactional_db`.
- `transactional_db` additionally gains **true SAVEPOINT semantics** (what its docstring already claims): session-scoped cloned engine, per-test connection + nested transaction + commit-restart listener, rollback on teardown. Its ~10 dependent fixtures keep their existing contracts.
- `tests/test_mariadb_portability.py` integration fixture: `create_all` → `command.upgrade` (the live-MariaDB CI job thereby proves the baseline's InnoDB DDL every PR; the errno-150 regression guard is preserved through the baseline path).
- End state: **zero `create_all` in tests** (enforced by the guard in §3).

### 6. Shared model registration (PR-1 carry-forward)
- New `register_all_models()` in `backend/orm/` — the single canonical "populate `Base.metadata`" entrypoint. Imported by `alembic/env.py`, `tests/test_mariadb_portability.py`, and `tests/conftest.py`, eliminating the copied-import-block drift flagged in PR-1's review.

### 7. Documentation touch-ups
- `CLAUDE.md`: update the stale "C5 … DEFERRED to MariaDB go-live" bullet to reflect the collapse.
- `README.md` dev-run section: note that startup migrates automatically (`RUN_MIGRATIONS_ON_STARTUP`), with the manual `alembic upgrade head` equivalent.

## Risks & mitigations
- **Autogen fidelity** (server defaults, JSON columns, enum rendering may autogenerate imperfectly): the baseline is hand-reviewed after autogen, and the empty-diff guard on BOTH dialects is the hard gate.
- **SAVEPOINT semantics** for `transactional_db` dependents: contained to one fixture; commit-restart listener pattern; full suite is the regression net.
- **Coverage delta**: deleting ~1.3k test lines with their covered code shifts the pool — gate stays ≥75%, verified before PR.
- **Render demo**: `e2e-sqlite` required check covers the new boot path pre-merge; post-merge live verification (boot, reseed, login, one write) closes local==GitHub==Render.
